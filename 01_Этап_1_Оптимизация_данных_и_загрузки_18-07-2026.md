# ЭТАП 1. ОПТИМИЗАЦИЯ ДАННЫХ И ЗАГРУЗКИ ЦИФРОВОГО ПРИЛОЖЕНИЯ

**Дата:** 18.07.2026  
**Статус:** техническое решение подготовлено; требуется внедрение в GitHub и Tilda.

## 1. Принятое решение

Для проекта оптимальна двухслойная модель:

### Редакторский слой

Владелец и контент-редактор продолжают изменять два простых файла:

- `data/materials.json`;
- `data/routes.json`.

Новая CMS и сервер не нужны.

### Технический слой

GitHub автоматически:

1. проверяет структуру и связи;
2. пересчитывает количество материалов по маршрутам;
3. собирает единый `dist/app-data.json`.

Страница `/navigator` загружает только этот единый runtime-файл через один общий
загрузчик. Все T123-блоки используют один Promise и больше не делают собственные
несогласованные запросы.

Это решение:
- не раздувает проект;
- сохраняет ручное редактирование JSON;
- устраняет рассинхронизацию двух файлов;
- готово к росту количества материалов;
- позволяет позже добавить поиск, задачи, теги и даты без смены архитектуры.

## 2. Что показал аудит production-данных

Проверено:

- **37 материалов**;
- **8 маршрутов**;
- JSON синтаксически корректен;
- дублирующихся `id` нет;
- ссылки имеют домен `mmalova.com`;
- ссылки на несуществующие маршруты отсутствуют.

Обнаружены семь групп предупреждений:

1. В маршруте `07` указано `count: 4`, фактически материалов **6**.
2. URL `/kniga/link26` используется тремя записями: `m025`, `m027`, `m028`.
3. У `m011` и `m027` пустой `status`.
4. Поле `desc` отсутствует у 35 из 37 материалов.
5. У `m005`, `m006`, `m008`, `m010`, `m013`, `m021`, `m022` стоит
   `expert: true`, но отсутствует `expert_name`.
6. У `m027` есть `expert_name`, но нет `expert: true`.
7. В `m013` вероятна опечатка: «тригеров» вместо «триггеров».

Критических ошибок, мешающих собрать единый файл, нет. Предупреждения не блокируют
первое внедрение, но должны быть закрыты перед полноценным глобальным поиском.

## 3. Что найдено в опубликованном коде `/navigator`

Три независимых блока загружают данные отдельно:

- дашборд — `routes.json` и `materials.json`;
- навигатор — повторно `routes.json` и `materials.json`;
- инструменты — еще раз `materials.json`.

Итого обычная загрузка создает **пять запросов данных**. При этом используются
разные URL и разные версии: `routes.json@main`, а `materials.json` — из
зафиксированного коммита.

Дополнительно в коде страницы остался локальный блок `@font-face`, хотя приложение
находится в том же проекте Tilda и должно использовать глобальные шрифты
«MM · Фундамент».

## 4. Почему GitHub Actions здесь нужен

GitHub Actions — не отдельный подрядчик, сервис или платная система. Это встроенная
автоматизация репозитория.

Практический смысл для владельца:

- ошибочный JSON не попадет в рабочую выдачу;
- не нужно вручную считать материалы в каждом маршруте;
- не нужно вручную объединять файлы;
- после настройки Tilda больше не правится при обычном обновлении каталога;
- в журнале GitHub видно, какой файл и на какой проверке не прошел.

## 5. Что такое `schema`, `scripts`, `snapshots`

### `schema`

Файлы правил. Они отвечают на вопросы:

- обязательное ли поле `title`;
- допустим ли пустой `status`;
- должен ли `route` иметь формат `01`;
- может ли `page` быть текстом;
- какие типы материалов разрешены.

Schema не заменяет и не дублирует материалы.

### `scripts`

Служебные программы, которые читают JSON, проверяют его и собирают
`dist/app-data.json`. Они написаны на Node.js без сторонних зависимостей.

### `snapshots`

Архив последних рабочих копий. На первом этапе отдельная папка не нужна:
ее функции уже выполняют история Git, браузерный кеш и встроенный fallback Tilda.
Это сознательное сокращение архитектуры.

## 6. Устойчивость в разных странах

После внедрения действует четыре уровня:

1. свежий кеш в браузере;
2. основной файл через jsDelivr;
3. резервный адрес GitHub Raw;
4. встроенная последняя рабочая копия внутри первого T123-блока.

Если оба внешних источника недоступны, пользователь все равно увидит каталог,
а не пустую страницу. При восстановлении связи кеш обновится.

Tilda остается платформой интерфейса, GitHub — редакторским хранилищем, jsDelivr —
ускорителем доставки, но ни один внешний источник не является единственной
точкой отказа.

## 7. Что подготовлено

В пакет входят:

- исходные production JSON;
- `routes.corrected.json` с автоматически исправленным count маршрута 07;
- два черновых schema-файла;
- валидатор;
- сборщик единого файла;
- workflow GitHub Actions;
- готовый `dist/app-data.json`;
- первый T123-блок единого загрузчика со встроенным fallback;
- точная инструкция замены трех фрагментов на `/navigator`;
- README редакторского процесса.

## 8. Задание владельцу — внедрение

### В GitHub

1. Создать папки `schema`, `scripts`, `dist`, `.github/workflows`.
2. Загрузить файлы из одноименных папок пакета.
3. В `data/routes.json` изменить count маршрута `07` с `4` на `6`
   либо заменить файл на `routes.corrected.json`.
4. Сделать commit в `main`.
5. Открыть вкладку **Actions** и проверить зеленый запуск
   `Validate and publish app data`.
6. Убедиться, что бот создал или обновил `dist/app-data.json`.

Если ветка `main` защищена от автоматической записи, шаг автокоммита не сработает.
Тогда сборка остается валидной, но `dist/app-data.json` нужно будет коммитить
вручную либо разрешить GitHub Actions запись в ветку.

### В Tilda

1. Добавить первым блоком `/navigator` новый T123.
2. Вставить `00_BAPP_DATA_LOADER_T123.html`.
3. В трех блоках заменить загрузку по инструкции
   `01_ПАТЧ_подключения_загрузчика.md`.
4. Удалить локальный `@font-face` из настроек страницы `/navigator`.
5. Опубликовать страницу.
6. Выполнить проверку из раздела 4 инструкции патча.

## 9. Контентные решения, которые можно закрыть после технического внедрения

Техническая миграция не зависит от ответов, но для следующего этапа нужно
зафиксировать:

- статусы `m011` и `m027`;
- авторов семи записей без `expert_name`;
- считать ли три записи `/kniga/link26` самостоятельными поисковыми результатами;
- если да — добавить точные якоря разделов и `parent_id`;
- постепенно заполнить `desc`;
- добавить `published_at` и `updated_at`, чтобы метрики «новое за 30 дней» и
  «обновления за квартал» вычислялись автоматически, а не вводились вручную.

## 10. Решение по schema

Ждать завершения большой schema не нужно. В пакет включен мягкий совместимый
черновик v0.1:

- текущие production-файлы проходят сборку;
- устаревшие неоднозначности показываются как предупреждения;
- критические ошибки блокируют публикацию;
- будущие поля уже предусмотрены, но пока необязательны.

После стабилизации данных черновик можно ужесточить без изменения Tilda-кода.
